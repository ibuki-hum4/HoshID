# AGENTS.md

Instructions for Codex and coding agents working in this repository.

## Language

すべてのユーザー向けの返答は日本語で行うこと。

対象:
- 説明
- 質問
- 作業計画
- 進捗報告
- コードレビュー
- 最終報告
- コミットメッセージ案

例外:
- ソースコード
- 識別子（変数名・関数名など）
- エラーメッセージやログの原文引用
- ユーザーが明示的に英語を要求した場合

技術用語は必要に応じて英語表記を併記してよいが、説明文は日本語で記述すること。

---

## 1. Think Before Coding

**推測しない。不明点を隠さない。選択肢とトレードオフを示す。**

実装前に:

- 前提条件や仮定を明示する。
- 要件が曖昧な場合は確認する。
- 複数の解釈が可能な場合は選択肢を提示する。
- より単純な方法がある場合は提案する。
- 不明なまま実装を始めない。

不確実な場合は推測より確認を優先すること。

---

## 2. Simplicity First

**要求を満たす最小限の実装を行う。**

原則:

- 要求されていない機能を追加しない。
- 単発利用のための抽象化を作らない。
- 将来のためだけの拡張性を追加しない。
- 不要な設定項目を増やさない。
- 新しい設計より既存パターンを優先する。
- 複雑さを増やすより減らす。

常に以下を確認する:

> この実装は経験豊富なエンジニアが見ても必要十分でシンプルか？

そうでないなら簡素化する。

---

## 3. Surgical Changes

**必要な箇所だけ変更する。**

既存コードを編集する場合:

- 関係ないリファクタリングを行わない。
- 関係ないコメントを書き換えない。
- 関係ないフォーマット変更を行わない。
- 必要のない命名変更を行わない。
- 既存のコードスタイルに合わせる。

関連しない問題を見つけた場合:

- 報告はしてよい。
- 勝手に修正しない。

自分の変更によって不要になったもののみ削除する:

- import
- 変数
- 関数
- 型定義

すべての変更行はユーザーの要求に直接対応していること。

---

## 4. Goal-Driven Execution

**成功条件を定義し、検証する。**

例:

- バグ修正
  - 再現手順を確認
  - 修正
  - 再現しないことを検証

- 新機能追加
  - テスト作成
  - 実装
  - テスト成功

- リファクタリング
  - 変更前後で挙動一致を確認

複数工程の作業では簡潔な計画を示す:

1. 作業内容
   - 検証方法

2. 作業内容
   - 検証方法

3. 作業内容
   - 検証方法

実装だけで完了と判断しない。

---

## Execution Workflow

単純な修正以外では以下を実施する:

1. 要求を理解する
2. 前提条件を整理する
3. 作業計画を示す
4. 成功条件を定義する
5. 最小限の変更を行う
6. テスト・ビルド・Lintを実行する
7. 成功条件を検証する
8. 結果を報告する

---

## Verification Requirements

作業完了前に可能な限り以下を実施する:

- テスト実行
- Lint実行
- 型チェック実行
- ビルド実行

実行できなかった場合は理由を報告する。

検証結果を確認せずに完了としない。

---

## Code Quality Rules

- 可読性を優先する
- 過度に賢い実装を避ける
- 不要な依存関係を追加しない
- 関数は単一責任を意識する
- 既存アーキテクチャを尊重する
- DRYより分かりやすさを優先する
- 小さく理解しやすい変更を好む

---

## Failure Handling

必要な情報が不足している場合:

1. 不足している情報を説明する
2. 確認事項を提示する
3. 回答を待つ

推測で実装しない。

存在しない仕様を作らない。
存在しないAPIを作らない。
存在しないライブラリ機能を作らない。

---

## Expected Completion Report

作業完了時は日本語で以下を報告する:

### 変更内容
- 実施した変更

### 変更理由
- なぜその変更が必要だったか

### 検証結果
- 実行したテスト・Lint・ビルド・型チェック

### 残課題
- 未解決事項や制約（あれば）

簡潔かつ事実ベースで報告する。
